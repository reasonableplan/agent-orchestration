## {{section_number}}. SLO / 성능 목표

### 응답시간 (Latency)

| 엔드포인트 / 액션 | p50 | p95 | p99 | 측정 방법 |
|-------------------|-----|-----|-----|----------|
| `<GET /resource>` | <50ms> | <200ms> | <500ms> | <서버 로그 / APM> |
| `<POST /heavy>` | <100ms> | <500ms> | <1s> | |
| `<background job>` | <N/A> | <N/A> | <30s> | (queue lag) |

### 가용성 (Availability)

| 항목 | 목표 | 월 허용 down time |
|------|------|-------------------|
| 전체 서비스 | <99.0% / 99.9% / 99.95%> | <7.2h / 43m / 21m> |
| 핵심 기능 (예: 결제) | <99.95%> | <21m> |
| 부수 기능 (예: 통계) | <99.0%> | <7.2h> |

### 처리량 (Throughput)

- **예상 RPS**: 평균 `<N>` / 피크 `<M>` (피크 시간대 `<HH:MM~HH:MM>`)
- **동시 사용자**: 평균 `<N>` / 피크 `<M>`
- **데이터 증가**: `<일 N row / 월 N GB>`

### 부하 프로파일

| 시나리오 | 빈도 | 처리 요구 |
|---------|------|----------|
| 평상시 | 24h | <목표 RPS 처리> |
| 일일 피크 | <HH:MM~HH:MM> | <피크 RPS x 2> 까지 처리 |
| 이벤트 / 마케팅 캠페인 | 월 N회 | <피크 RPS x 5> 까지 (사전 스케일업) |

### Error budget

- **월 error budget**: `(1 - 가용성 목표) × 30일` = `<예: 99.9% → 43분>`
- **소진 시 정책**: 신규 기능 배포 동결, 안정화에 집중
- **갱신 주기**: 매월 1일

### SLO 위반 시 액션

- **자동 페이지**: p99 > <목표 × 2> 가 5분 이상 지속 → 온콜 페이지
- **자동 스케일**: CPU > 70% 또는 큐 길이 > <임계> → 인스턴스 +1
- **자동 차단**: 단일 IP RPS > <임계> → rate limit (rate_limiting 정책 참조)

> 작성 가이드:
> - p99 는 "사용자가 가장 늦게 받는 응답" — UX 결정짐
> - 가용성 목표는 비즈니스 비용과 균형 — 99.99% 는 multi-region 필요 (capacity 비용 폭증)
> - 부하 프로파일은 실측 또는 비즈니스 가정에 근거. "그냥 충분히" 금지
> - SLO 위반은 `runbook` 의 입력 — 알람 + 대응 절차로 연결
> - 6축 `availability == high` 면 SLO 단순 정의 외에 multi-region / DR 도 필수
